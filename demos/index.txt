# Demos 

Experimentation is about learning lessons and sometimes the failures as important as the successes.
A perfect companies demo page would highlight their endless successes and unmatched expertise. Yet 
we all know that in the pusuit of excellance, we are going to make mistakes. Some might say,  
if you don't fail, if you don't make mistakes, it is becuase you have not tried anything new. Others 
might say if you don't fail, you are not even trying.

This is what make Bearzneker different. Our business model allows us to fail on our time, not your
dime. We make controlled mistakes, document our failures, and present them as lesson learned.

And we have fun doing it. That's what this demo page is about. Fun experiments that push our  
boundaries. Not to show you how we are perfect, but to show you we are ready.

Each of these experiments had a specific goal. The goals were purposely complicated with significant 
opportunity for failure, but they all recovereed into a minimal viable product. Enjoy.

## Conquest
The Conquest demo was created using the Google Antigravity V2 coding agent, backed by Gemini 3.1 Pro.

Antigravity V2 is a departure from the traditional coding agent. As of July 2026, Google has introduced three code agents:
- Command Line Agent: Simlilar to Claude Code, it ran on a local code base and performed tasks as directed.
- Jules Cloud Agent: Ran entirely from a GCP sandbox by cloning a remote Git repo.
- Antigravity V1: An IDE with agentic code creation, which may or may not have been a fork 
of VS Code with a Gemini plugin.

Antigravity V2 infuriated Internet devs by taking away the IDE. The application's operational model was "tell me what you want, and I'll make it happen". For Internet devs, this felt too much like "vibe coding". In the collective mind of these very vocal users V2 would empower workers who do not understand programming to create unsupportable "slop code".

The purpose of this experiment was to evaluate if the code was actually unsupportable. Eventually all professional devs find themselves having to add a feature to a code base they do not understand. 
This has always been a significant pain point. But the root cause of this pain is that dev A wrote 
the code, has left, and now dev B has to reverse engineer the product.

What Antigravity V2 maintained a history of all of its actions and provided a way for each Gemini 
iteration to be aware of its predecessor's thought processes? If this were the case, each feature add 
is not a case of divining your way through someone else's speghetti code, but instead is just 
another Agile sprint.

This was the approach that was taken with "Conquest".
- An initial prompt was provided to define the overall objective.
- The agent was given full autonomy to define look and feel.
- Five sprints defined features and bug fixes.
- Two hours was spent on the project.

The outcome is presented is a minimailly functional game, without a single line of code being written.
* Remember: the game was not the goal, so no current bugs will be fixed.

The sprint methodology performed as expected, but this experiment raises an important philosophical question: Do "vibe coders" understand Agile or Scrum enough create a discriplined workflow?

The obvious answer is no, but "we" do.

The general argument against vibe coding is that people with limited understanding are generating code from a prompt, with limited (or no) understanding of what the code does. This is the same complaint that was levied against "cargo cult" development, where a person copied a block of code from Stack Exchange and pasted it into a project. Once accused of being a "cargo cult" development, a person was condemmed to perpetual side-eye from real "console cowboys".

Yet over the last decade very few shops has experienced a meltdown from cut and paste development. If the Stack Exchange code blocks are thought of as Legos, the "cargo cult" developer becomes an assembler of standardized parts. A hobbiest that makes Lego models does not need to know how to extrude the indiviual blocks to construct a completely unique model. If we brush aside the idea that vibe coding is dangerous because the creator does not understand the creation, we are back to the argument of unsupportability. 

This experment demonstrated that Antigravity V2 resolves the idea of unsupportability by:
- Opening a local folder for each application project.
- Creating a chat list to act as memory of actions to the project.
- Downloading "reasoning" to the project.

As changes are requested, a new chat is created. Each chat represents a sprint. The agent is given instructions:
```
After "End Attack Phase", the players should get an army movement phase. It will work like the challenge phase, where the player will choose a territory they occupy. Instead of highlighting opponent territories (red dashed circle), it will highlight the player's connected teritories (green dashed cirle). A destination will be chosen. A popup will show the number of armies and give a +/- to move armies over, leaving at least one behind. There will be a "End Movement Phase" button.  (AI players will have the same movement capability)
```
The agent's thought process (reasoning) is commited to the project:
```
I'm now deeply considering a new phase post-attack: a Fortify/Movement Phase. Players would get to move units and set up defenses after attacks conclude, mirroring attack phase mechanics. This allows for tactical repositioning, counter-maneuvering, and strategic resource allocation after battles, adding another layer of depth to the strategy.

<snip>
```
An implentation plan is created:
```
I have created the implementation plan for the new Army Movement Phase (Fortification Phase). It outlines:

Visual highlights for connected owned territories (glowing green dashed circles).
The design of a custom glassmorphic slide-control modal with increment/decrement buttons.
Personalized AI movement algorithms (Cautious defending, Wreckless doomstacking, and Strategic border contesting).
Full state machine transition logic.
Please review the implementation plan and let me know if you approve so I can begin coding this feature!
```
In theory, every future change has access to every past decision.
